## 🔴 BUG CRÍTICO #1 — Botão CONCLUIR não registra receita

**Severidade:** 🔴 CRÍTICO — Bloqueador de produção
**Arquivo:** `pages/admin/Agenda.tsx` · função `handleActionComplete` (linha ~400)
**Módulos afetados:** Agenda ↔ Financeiro ↔ Dashboard

### Descrição
Quando o usuário clica em **CONCLUIR** (via modal de ações ou via comanda), o sistema apenas
atualiza o status do agendamento para `'done'` no banco. Nenhum registro é inserido na tabela
`financial_records`. Resultado: a receita do atendimento **nunca aparece** no financeiro nem
nos gráficos do Dashboard.

### Evidência
```sql
-- Após marcar 3 agendamentos como 'done' (R$35 + R$25 + R$55 = R$115):
SELECT COUNT(*) FROM financial_records
WHERE appointment_id IN ('fff...001', 'fff...002', 'fff...003');
-- Resultado: 0
```

### Código problemático
```typescript
// Agenda.tsx - handleActionComplete (atual)
const handleActionComplete = async () => {
    if (!selectedAppointment) return;
    await supabase.from('appointments').update({ status: 'done' }).eq('id', selectedAppointment.id);
    // ❌ FALTA: INSERT em financial_records
};
```

### Correção necessária
```typescript
const handleActionComplete = async () => {
    if (!selectedAppointment || !selectedMatriz) return;

    // 1. Marcar agendamento como done
    await supabase.from('appointments')
        .update({ status: 'done' })
        .eq('id', selectedAppointment.id);

    // 2. Buscar preço do serviço
    const { data: serviceData } = await supabase
        .from('services')
        .select('price, name')
        .eq('id', selectedAppointment.service_id)
        .single();

    // 3. Inserir em financial_records
    if (serviceData) {
        await supabase.from('financial_records').insert({
            matriz_id: selectedMatriz.id,
            type: 'income',
            amount: serviceData.price,
            description: `Atendimento: ${serviceData.name} — ${selectedAppointment.client}`,
            date: new Date().toISOString().split('T')[0],
            appointment_id: selectedAppointment.id,
            payment_method: comandaPaymentMethod === 'money' ? 'Dinheiro'
                           : comandaPaymentMethod === 'credit' ? 'Crédito' : 'PIX',
            payment_status: 'paid',
            client_name: selectedAppointment.client
        });
    }

    setAppointments(prev =>
        prev.map(a => a.id === selectedAppointment.id ? { ...a, status: 'done' } : a)
    );
    setIsActionModalOpen(false);
    setSelectedAppointment(null);
};
```

**Alternativa (mais robusto):** Criar trigger no Supabase:
```sql
CREATE OR REPLACE FUNCTION auto_create_financial_record()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  IF NEW.status = 'done' AND OLD.status != 'done' AND NEW.service_id IS NOT NULL THEN
    INSERT INTO financial_records (matriz_id, type, amount, description, date, appointment_id, payment_status)
    SELECT NEW.matriz_id, 'income', s.price,
           'Atendimento: ' || s.name, CURRENT_DATE, NEW.id, 'paid'
    FROM services s WHERE s.id = NEW.service_id;
  END IF;
  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_appointment_done
  AFTER UPDATE ON appointments
  FOR EACH ROW EXECUTE FUNCTION auto_create_financial_record();
```

## 🔴 BUG CRÍTICO #2 — Saldo Previsto sempre R$0,00

**Severidade:** 🔴 ALTO — Feature não funcional
**Arquivo:** `pages/admin/Dashboard.tsx` · função `generateDataForFilter` (linha ~328)
**Módulos afetados:** Agenda ↔ Dashboard (gráficos financeiros)

### Descrição
Os campos `entradasPrev`, `saidasPrev` e `saldoPrev` estão **hardcoded como `0`** em todo o
código. O Dashboard nunca consulta agendamentos futuros para calcular o saldo previsto. Com
R$165,00 em agendamentos futuros confirmados/pendentes no banco, o gráfico exibe **R$0,00 previsto**.

### Evidência
```typescript
// Dashboard.tsx - generateDataForFilter (atual)
return { name: pad(day), entradas, saidas: -saidas, saldo: entradas - saidas,
         entradasPrev: 0,   // ❌ HARDCODED
         saidasPrev: 0,     // ❌ HARDCODED
         saldoPrev: 0 };    // ❌ HARDCODED
```

```sql
-- 3 agendamentos futuros existem com receita total de R$165:
SELECT SUM(s.price) AS receita_prevista FROM appointments a
JOIN services s ON a.service_id = s.id
WHERE scheduled_at > NOW() AND status IN ('pending','confirmed');
-- Resultado: 165.00 → o Dashboard exibe: 0,00
```

### Correção necessária
No `useEffect` do Dashboard, fazer fetch adicional de appointments futuros:
```typescript
// Buscar receita prevista (agendamentos futuros pendentes/confirmados)
supabase
  .from('appointments')
  .select('scheduled_at, services(price)')
  .eq('matriz_id', selectedMatriz.id)
  .in('status', ['pending', 'confirmed'])
  .gt('scheduled_at', new Date().toISOString())
```
E usar esses dados para popular `entradasPrev` no `generateDataForFilter`.

## ⚠️ BUG MÉDIO #3 — Agendamentos cancelados reaparecem após reload

**Severidade:** ⚠️ MÉDIO — UX degradada, causa confusão
**Arquivo:** `pages/admin/Agenda.tsx` · função `fetchAppointments` (linha ~225)
**Módulos afetados:** Agenda (calendário semanal)

### Descrição
A função `handleActionCancel` remove o agendamento do estado local com `.filter()`, mas a
query `fetchAppointments` não exclui `status = 'cancelled'`. Ao trocar de semana ou recarregar
a página, todos os agendamentos cancelados reaparecem no calendário.

### Evidência
- `handleActionCancel`: faz `UPDATE status='cancelled'` + `setAppointments(prev => prev.filter(...))`
- `fetchAppointments`: não tem `.neq('status', 'cancelled')` na query do Supabase

### Correção (1 linha)
```typescript
// Agenda.tsx - fetchAppointments
const { data, error } = await supabase
    .from('appointments')
    .select(`...`)
    .eq('matriz_id', selectedMatriz.id)
    .neq('status', 'cancelled')  // ✅ ADICIONAR ESTA LINHA
    .gte('scheduled_at', start.toISOString())
    .lte('scheduled_at', end.toISOString());
```
